总结

  1. setnx 只有当key不存在时才能成功，返回1。否则失败返回0

    setnx 无法同时set时间，所以会出现set有效时间，失败的问题。

  2. 如果value值相同时，假如执行时间超过锁的超时时间，当别的线程又获得锁的时候，当前线程会将其他线程的锁给释放，所以value值应当使用不同的值。

    新的问题是，释放锁的代码线程不安全。但是这已经能解决普遍的分布式锁的问题了，因为要出现在线程不安全的情况，需要同一个value值，出现在多线程中

  3. 通过lua脚本，将对比与释放包装成原子操作

setnx

主要是利用SETNX 只有在 key 不存在时设置 key 的值才能成。成功时返回1，失败时返回0

  • redis 的三种写法
  • redis 锁固定的值，这种是肯定不行的，如上所示，要是出现线程3的情况，任务处理时间超过加锁的有效时间，线程3删除锁的时候，很有可能会把其他线程的锁删除。
  • redis 锁不同的值，这种使用方式的担忧，往往是因为finally语句块中的代码是线程不安全，其实对于单个的锁定值（idStr） ，finally 语句块中出现并发的业务场景确实不多。
    假如真的出现了（比如线程一里面，try语句块+ boolean isLock = false; 再出现并发，这也还是变成了问题一的写法，不同的线程锁同一个value，那为什么不能锁不通的value呢），如果 finally语句块中没有 if判断，确实会出现线程不安全的问题。但是加上 if 判断，就能够保证 unlock不出现并发。
    其实这种写法已经能够解决日常开发中大部分情况了。
