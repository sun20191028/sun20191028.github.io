filter 与 Interceptor的区别

  • Interceptor 的原理是基于动态代理-由spring mvc 引入，而filter 是基于链式模式的，由tomcat等web容器引入
  • Interceptor 不拦截静态资源

spring mvc xml 添加 listener、filter、servlet、Interceptor

在application.xml中配置

java config + listener、filter、servlet、Interceptor

  1. 添加外部的 listener、filter、Interceptor
  1. 添加自定义的 listener、filter

一个WebContent 就是tomcat中的一个 StandardContext 。与tomcat自带的 examples、root、host-manager、manager同属于一个StandardHost

Listener的定义与作用

监听器Listener就是在application,session,request三个对象创建、销毁或者往其中添加修改删除属性时自动执行代码的功能组件。
　　Listener是Servlet的监听器，可以监听客户端的请求，服务端的操作等。

listener触发时机

context 的创建在系统启动的时候，适合做连接池初始化，context销毁时做连接池关闭的动作
context 的属性变化没什么监听的

session的创建，通常在第一次请求经后台跳转到jsp页面的时创建（jsp默认开启session），session销毁一般等超时，或手动调用invalidate()
session属性变化，直接在session.setAttribute的时候直接触发

request 的创建，在请求封装request的时候触发

Listener的分类

ServletContext监听

ServletContext的初始化和销毁

ServletContextListener：用于对Servlet整个上下文进行监听（创建、销毁）。

ServletContextAttribute增加、修改、删除

ServletContextAttributeListener：对Servlet上下文属性的监听（增删改属性）。

Session监听

Session属于http协议下的内容，接口位于javax.servlet.http.*包下。

  • session的销毁有两种情况：
    1. session超时，web.xml配置：
    2. 手工使session失效

HttpSession 的创建、销毁

HttpSessionListener接口：对Session的整体状态的监听。

HttpSessionAttribute增加、修改、删除

HttpSessionAttributeListener接口：对session的属性监听。

Request监听

ServletRequest的创建和销毁

ServletRequestListener：用于对Request请求进行监听（创建、销毁）。

ServletRequestAttribute增加、删除、修改

ServletRequestAttributeListener：对Request属性的监听（增删改属性）。

在web.xml中配置

Listener配置信息必须在Filter和Servlet配置之前，Listener的初始化（ServletContentListener初始化）比Servlet和Filter都优先，而销毁比Servlet和Filter都慢。
